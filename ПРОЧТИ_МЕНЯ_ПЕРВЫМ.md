# SilentConnect — что это и как тут всё устроено

Этот файл — точка входа для любого нового чата с ИИ (Claude, Codex, Gemini, Antigravity — неважно).
Прочти его целиком перед тем, как что-то менять на серверах. Здесь собрано всё, что раньше
приходилось объяснять заново в каждом новом чате.

## Что за проект

SilentConnect — коммерческий VPN-сервис (перепродажа доступа). Технически: 3X-UI + Xray-core +
собственный Flask-сервис `subjson-service`, который генерирует подписки для клиентского приложения
**Happ** (Windows/Android/iOS). Продажи и поддержка клиентов — через Telegram-бота `vpn-shop`.

У каждого клиента в приложении Happ есть 5 профилей подписки на каждый сервер:
1. **🛡️ Классический** — VLESS + TCP + REALITY (SNI = sber.ru)
2. **⚡ Быстрый** — VLESS + TCP + REALITY (SNI = st.kinopoisk.ru), тот же порт 443, тот же мультиплексор
3. **🚀 Скоростной** — Hysteria2 (UDP/QUIC), порт 1443
4. **🔐 Запасной** — VLESS + gRPC + REALITY (SNI = vk.com), отдельный порт 29443
5. **🌊 Незаметный** — VLESS + XHTTP (HTTP/3), отдельный порт

## Архитектура на каждом сервере (NL/FI одинаковая)

- **3X-UI** (форк mhsanaei) — панель управления, но реально управляет только частью инбаундов.
- **Отдельный systemd-сервис `xray-maxru`** — свой статический конфиг
  `/usr/local/etc/xray-maxru/config.json`, обслуживает все 5 профилей. Порт 443 — это
  `dokodemo-door` инбаунд с тегом `multiplexer-443`, который снифает SNI и решает: известный
  домен (сайты самого SilentConnect) → `to-caddy`, всё остальное → `to-reality` (REALITY на
  порту 24443, мультиплексирует Классический/Быстрый по SNI).
- **Caddy** слушает на 4430 (НЕ на 443 — 443 занят dokodemo-door), проксирует на
  `subjson-service` (порт 3088).
- **`subjson-service`** (`/root/subjson-service/app.py` на сервере) — генерирует JSON-подписку
  для Happ с нужными заголовками (`tun-enable`, `providerid` и т.п.).
- **Happ** — клиент, на Windows использует свой xray.exe
  (`C:\Program Files\FlyFrogLLC\Happ\core\xray.exe`) и TUN через sing-box (`happ-tun`).

## Статус на сегодня (2026-07-04)

**NL** — полностью рабочий, используется как референс.

**DE** — временный сервер, клиентам разослано официальное уведомление о переходе на NL
до отключения DE. Стоит на очереди на полное выключение.

**FI** — 4 из 5 профилей были нерабочими для реальных клиентов ("подключено, но без интернета").
За эту сессию **исправлено и подтверждено**:
- Отсутствовал systemd-сервис `xray-ws443` (WiFi-компаньон профиль) — создан, работает.
- IPv6 был выключен на FI, включён на NL — синхронизировано (позже выяснилось, что оба сервера
  реально имеют IPv6 выключенным на уровне sysctl — расхождение было не в этом).
- В `subjson-service/app.py` профиль Hysteria2 ("Скоростной") был жёстко прописан на порт 443
  вместо 1443 — исправлено в двух местах, проверено на реальном трафике (tcpdump + access.log
  показали доступ к Telegram/Spotify через этот профиль).
- Неправильная подпись (`serverDescription`) у профиля "Незаметный" — исправлена.

**Ещё не закрыто на конец сессии**: обнаружена воспроизводимая проблема — при подключении к FI
через SNI, отличный от `sber.ru` (то есть `vk.com`, `st.kinopoisk.ru`, и вообще любой домен,
включая случайные типа `microsoft.com`), соединение на порту 443/24443 **зависает без ответа**,
если идти с внешнего IP. При этом:
- С `localhost` самого сервера FI — вообще все SNI работают нормально.
- На NL (той же конфигурацией, тем же REALITY, тем же `target: vk.com:443`) — все SNI работают
  нормально и с внешнего IP тоже.
- Версия xray-core была временно уравнена с NL (26.3.27) — не помогло, проблема не в версии.
- IPv6 одинаково выключен на обоих серверах — не в этом.
- На сервере FI нет никаких firewall/fail2ban правил, блокирующих тестовый IP.

Рабочая гипотеза на конец сессии: похоже на реакцию сетевой инфраструктуры Hetzner (дата-центр,
не РКН/провайдер) на аномально интенсивное тестирование одного IP с разными SNI подряд —
трактуется как сканирование. Требует проверки после паузы в тестировании и/или обращения в
техподдержку Hetzner. **Подробности и следующие шаги — в `документация_и_архитектура/`**,
ищи файлы, обновлённые последними по дате.

## Жёсткие правила работы (важно!)

1. **Не трогать личные устройства пользователя** (телефон, ПК, настройки Happ) для диагностики,
   пока не исчерпаны ВСЕ серверные варианты. Причина: в прошлых сессиях (Antigravity/Codex)
   теория "дело в устройстве/провайдере/РКН" **каждый раз** оказывалась неверной — реальная причина
   всегда была на сервере (заголовки, инбаунды/аутбаунды, UUID, нюансы базы 3X-UI). Единственное
   легитимное исключение было один раз — реальная ошибка маршрутизации Windows TUN.
2. Перед тем как поверить результату локального теста через `xray.exe` — убедись, что тестовый
   процесс реально запустился на СВОБОДНОМ порту (не 10808/10809 — их может держать сам живой Happ).
   В этой сессии половина "аномальных" результатов оказалась именно этим: тест тихо не стартовал,
   а curl уходил в реально работающее соединение Happ.
3. На сервере FI сейчас 71+ реальных активных клиентов — логи `xray-maxru` содержат ИХ трафик
   вперемешку с твоими тестами. Сверяй source IP теста с тем, что реально видно в логе, не
   полагайся на совпадение по времени.
